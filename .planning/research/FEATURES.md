# Feature Landscape: Chart Generation & Data Visualization

**Domain:** AI CSV analytics — inline chart rendering from natural language queries
**Researched:** 2026-04-20
**Milestone scope:** CHART-01 through EXPORT-02 (adding charts to an existing working RAG chat loop)

---

## How Competing Products Handle Chart Generation

### ChatGPT Advanced Data Analysis
**Approach:** Python code execution in a sandboxed REPL. The model writes matplotlib/pandas code, executes it server-side, and returns a rendered PNG image embedded in the chat thread.
**Export:** PNG image download. No SVG. No CSV re-export from chart.
**Interactivity:** None — static image only. No tooltips, no zoom.
**Chart selection:** Model decides based on data shape and prompt. No user override UI.
**Key observation:** Code execution gives maximum flexibility but is opaque to users and requires a sandboxed execution environment — not viable for a Next.js SaaS.

### Julius AI
**Approach:** Natural language → chart generation via Python (matplotlib/seaborn) executed server-side. Charts appear inline in the chat thread as images.
**Chart types:** Bar, line, scatter, pie, histogram, box plot, map, treemap, Gantt, 3D plots.
**Export:** Right-click to download; CSV/Excel export on request via chat; download link in chat thread.
**Interactivity:** Interactive plots (hover/zoom) for some chart types; static PNG for others.
**Key observation:** Julius's real differentiator is automatic date normalization and format-handling, not its chart UX. The image-in-chat pattern is functional but not polished.

### Rows.com
**Approach:** Spreadsheet-native. AI Analyst generates charts attached to the spreadsheet, not free-floating in a chat. Charts are first-class spreadsheet objects.
**Chart types:** Bar, line, combo (bar+line), stacked area, and all standard spreadsheet chart types.
**Export:** CSV/XLSX export. Chart export via spreadsheet export.
**Interactivity:** Full spreadsheet chart interactivity.
**Key observation:** Rows is a spreadsheet tool, not a chat tool. The mental model is different — chart is a persistent artifact, not a conversational response. Not a direct comparator.

### Vanna.ai
**Approach:** Text-to-SQL → data → Plotly or Vega-Lite spec generated by LLM from data + SQL + question. Charts rendered as rich components in chat (not images).
**Chart types:** Whatever Plotly supports (extensive).
**Export:** Not standardized — depends on embedding.
**Architecture insight:** LLM generates a chart spec (Plotly JSON or Vega-Lite), not Python code. Spec is passed to a renderer. This is the closest pattern to what this project uses.
**Key observation:** The Vanna/chat2plot approach — LLM emits a declarative JSON spec, renderer converts to visible chart — is the right pattern for a Next.js SaaS. No sandbox required, spec is inspectable, rendering is deterministic.

### chat2plot (open-source library)
**Approach:** LLM function-calling API → structured JSON chart definition (via Pydantic schema) → rendered with plotly/altair.
**Advantage over code execution:** More secure (no arbitrary code), language-agnostic, user-editable spec, lower error rates (~3.5% in single-shot, ~0% with repair).
**Key observation:** Confirms that structured JSON tool-call approach is the industry-accepted pattern for safe, reliable chart rendering without a Python sandbox.

## Standard Approach: JSON Tool Calls vs Code Execution vs Markdown

| Approach | Security | Reliability | Flexibility | Complexity | Right for This Project |
|----------|----------|-------------|-------------|------------|----------------------|
| Python code execution (ChatGPT style) | Low (sandbox required) | Medium (hallucinated code errors) | Highest | High (sandbox infra) | No |
| Structured JSON tool call (Vanna/chat2plot style) | High | High (~0% with validation) | Medium (schema-bounded) | Low | **Yes** |
| Markdown/text instruction | N/A | Low (parsing fragile) | Low | Low | No |
| Image return (Julius style) | High | High | Low (not interactive) | Medium | No |

**Verdict:** Structured JSON tool call is the correct approach for this project. The AI emits a `chart_spec` tool call with typed fields (`type`, `x_column`, `y_column`, `title`, `data`). The `<ChartBlock>` Client Component renders it with Recharts. This is already the project's stated direction (PROJECT.md) and is validated by Vanna.ai and chat2plot.

## Table Stakes

Features users expect. Missing = product feels incomplete or broken.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Bar chart | Most common chart type for categorical comparison; every competitor has it | Low | Recharts `<BarChart>` |
| Line chart | Standard for time-series trends; CSV data is often time-series | Low | Recharts `<LineChart>` |
| Pie chart | Expected for proportional/composition data; users will ask "what percentage" questions | Low | Recharts `<PieChart>` — use sparingly, but users expect it |
| Scatter chart | Expected for correlation questions ("is X related to Y?") | Low | Recharts `<ScatterChart>` |
| Chart appears inline in chat thread | Users expect answers and visualizations together, not on a separate page | Medium | `<ChartBlock>` rendered inside the message bubble |
| Chart has a title | Minimum legibility requirement; untitled chart feels broken | Low | From `chart_spec.title` field |
| Axis labels on bar and line charts | Without labels, axes are meaningless | Low | From `x_column`/`y_column` field names |
| Tooltip on hover | Standard interaction for every web chart; users hover instinctively | Low | Recharts built-in `<Tooltip>` |
| Chart renders within chat column width | Chart that overflows its container looks broken | Low | `ResponsiveContainer` with `width="100%"`, fixed height |
| AI text answer accompanies the chart | Chart alone without explanation is insufficient — users need the "so what" | Medium | Chart is additive to, not a replacement for, the text answer |
| Chart only appears when data warrants it | Showing a chart for a question like "what is the mean salary?" when a number suffices frustrates users | Medium | `chart_spec` tool call is optional; AI decides when to emit it |
| PNG export/download | Users will want to paste chart into a report or slide; standard expectation across all competitors | Medium | `<canvas>` or `html-to-image` conversion |

## Differentiators

Features that provide competitive advantage for this product. Not expected by default, but clearly valued.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| AI narrates the chart insight | "Revenue peaked in Q3, likely driven by…" appears below the chart — not just the chart alone. Most competitors show a chart without a sentence-level insight | Low | Add `insight` field to `chart_spec` schema; render as caption |
| Chart type picker (post-render) | Small button strip (bar / line / pie) to re-render with a different type without re-prompting. Addresses the "wrong chart type" frustration identified in GoodData pitfalls research | Medium | Client-side re-render from same data; no new AI call |
| Clipboard copy (not just download) | "Copy image" is faster than download for pasting into Slack/Notion/email. Julius does right-click only; a dedicated button is better UX | Low | `navigator.clipboard.write()` with Blob |
| Graceful degradation message | If chart spec is invalid or data is insufficient, show a friendly fallback ("Not enough data to chart — here's the raw answer") instead of blank or error | Low | Error boundary around `<ChartBlock>` |
| Data table toggle | Show/hide the raw data rows that the chart is based on, alongside the chart. Builds trust ("I can verify the AI's data") | Medium | Collapsible `<DataTable>` below chart, same data from spec |

## Anti-Features

Features to explicitly NOT build in v1.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Python code execution / REPL sandbox | Requires sandboxed execution infra (Docker, Pyodide, etc.). High complexity, security surface, maintenance burden. Not needed when structured JSON tool calls work reliably | Use `chart_spec` JSON tool call schema |
| Full Plotly / D3 / Vega-Lite integration | All are overkill for 4 chart types. Plotly alone adds ~1MB. Recharts already in the project via shadcn | Stick with Recharts for the 4 required types |
| Chart builder / drag-and-drop editor | Rows.com and Tableau do this. It requires a fundamentally different product surface (grid, column picker, encoding panel). Breaks the chat-first mental model | Let AI generate the spec; offer a type-switcher button only |
| Real-time / streaming chart data | No use case in CSV analysis — data is static file | Ignore |
| 3D charts (3D bar, 3D scatter) | Poor accessibility, rarely improves insight, users don't ask for them in CSV analysis, adds rendering complexity | Refuse in prompt, return 2D equivalent |
| Histogram, box plot, heatmap in v1 | Valid chart types but address long-tail use cases. Users will ask for bar/line/pie/scatter 95% of the time. Complexity is not worth it for v1 | Add in a future iteration if users ask |
| PDF export | Heavy dependency (jsPDF, Puppeteer, etc.), formatting complexity. Users who want PDF can print-to-PDF from browser | PNG + clipboard is sufficient for v1 |
| SVG export | Valid technically, but PNG is what users actually use for pastes/reports. SVG export requires additional tooling | PNG only for v1 |
| Chart saved to a "dashboard" | Requires a separate persistence layer, grid layout, saved state. Personal tool scope doesn't need this | Charts live in the chat thread; chat history is the "dashboard" |
| Chart annotations / drawing tools | Editing/annotation requires a canvas editor. Out of scope for a chat-first product | |
| Multiple charts per message | Cognitive overload. If AI emits multiple chart specs, render only the most relevant one | Constrain `chart_spec` tool call to at most 1 per response in the system prompt |

```
chart_spec tool call schema defined
  → ChartBlock Client Component
      → Bar/Line/Pie/Scatter rendering (CHART-03)

      → Tooltip (Table Stakes)
      → ResponsiveContainer sizing (Table Stakes)
      → Chart type switcher (Differentiator — depends on ChartBlock)
      → Data table toggle (Differentiator — depends on ChartBlock)
      → AI insight caption (Differentiator — depends on chart_spec schema)

ChartBlock renders successfully
  → PNG export via html-to-image (EXPORT-01)
  → Clipboard copy (Differentiator)
```

## MVP Recommendation

Prioritize in order:

1. **`chart_spec` tool call schema** — Define the JSON schema the AI returns. Fields: `type` (bar|line|pie|scatter), `x_column`, `y_column`, `title`, `data` (array of row objects), `insight` (optional string). This is the foundation everything else builds on.
2. **`<ChartBlock>` Client Component** — Renders all four chart types from a spec. Uses Recharts `ResponsiveContainer`. Includes title, axis labels, and tooltip. Falls back gracefully on invalid spec.
3. **All four chart types** — Bar, line, pie, scatter are equally required. No subset is acceptable.
4. **Chart inline in chat** — Render `<ChartBlock>` inside the AI message bubble, below the text answer.
5. **PNG download** (EXPORT-01) — One button, `html-to-image` or canvas export. Required before v1.0 ships.

Defer to next milestone:
- **Chart type switcher** — Valuable differentiator but not blocking. Add once ChartBlock is stable.
- **Data table toggle** — Nice trust-builder, not blocking.
- **Clipboard copy** — Polish feature. Add alongside or after download.
- **Markdown/TXT export** (EXPORT-02) — Straightforward once PNG is done, but lower priority.

## Recharts v3 Compatibility Note

Recharts reached v3 stable (v3.7.0 as of research date). Key facts for this project:

- **React 19 support:** Confirmed. React 16.8+ supported; React 19 works in practice.
- **ResponsiveContainer:** Use `width="100%"` and a fixed `height` (e.g., `300` or `350`). Do NOT use percentage height — it requires a parent with explicit height.
- **Known issue:** `ResponsiveContainer` requires a parent `div` with a defined width to measure correctly. Do not render inside a flex container without explicit width.
- **Breaking changes in v3:** Internal state props removed (`CategoricalChartState`, `activeIndex`, `Legend.payload`). Custom `Customized` wrapper no longer passes internal state. Avoid relying on these.
- **Pie Chart:** `activeShape`/`inactiveShape` props deprecated in v3. Use `isActive` callback pattern instead.
